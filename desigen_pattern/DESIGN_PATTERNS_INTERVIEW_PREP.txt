1) FACTORY PATTERN
======================================================================

Code:
- desigen_pattern/1_factory/factory_desigen_pattern.py

One-line definition:
- Factory Pattern creates objects without exposing concrete creation logic to the client.

When to use:
- object creation is repetitive or complex
- client should not know concrete classes
- new product types may be added later

Real-world example:
- notification system creates Email, SMS, or Push sender

In your code:
- Product interface: Notification
- Concrete products:
  - EmailNotification
  - SMSNotification
  - PushNotification
- Factory: NotificationFactory

Most frequently asked questions:

Q1. Why use Factory instead of creating objects directly?
Answer:
- It hides creation logic
- reduces tight coupling
- makes code easier to extend

Q2. What problem does Factory solve?
Answer:
- It removes object creation decisions from the client
- the client works with abstractions, not concrete classes

Q3. What is the difference between Simple Factory and Factory Method?
Answer:
- Simple Factory: one class creates objects using conditions or mapping
- Factory Method: subclasses decide which object to create

Best interview line:
"Factory Pattern centralizes object creation so the client remains simple and
does not depend directly on concrete classes."

2) SINGLETON PATTERN
======================================================================

Code:
- desigen_pattern/2_singleton/singleton_desigen_pattern.py

One-line definition:
- Singleton ensures that only one instance of a class exists in the application.

When to use:
- logger
- config manager
- cache manager
- database connection manager

Real-world example:
- shared application configuration object

In your code:
- Singleton class: AppConfig
- single instance stored in `_instance`
- thread safety handled using lock

Most frequently asked questions:

Q1. Why not just use a global variable?
Answer:
- Singleton gives controlled access
- object creation stays inside the class
- lifecycle is better encapsulated

Q2. What is the drawback of Singleton?
Answer:
- behaves like global state
- can make testing and dependency management harder

Q3. Why is thread safety important in Singleton?
Answer:
- multiple threads may create multiple instances at the same time
- lock prevents race condition

Best interview line:
"Singleton is useful when exactly one shared instance should exist across the system."

3) STRATEGY PATTERN
======================================================================

Code:
- desigen_pattern/3_strategy/strategy_desigen_pattern.py

One-line definition:
- Strategy Pattern defines interchangeable algorithms and allows switching them at runtime.

When to use:
- multiple ways to do the same task
- behavior should change dynamically
- want to remove long if-else chains

Real-world example:
- payment with UPI, credit card, or debit card

In your code:
- Strategy interface: PaymentStrategy
- Concrete strategies:
  - UPIPayment
  - CreditCardPayment
  - DebitCardPayment
- Context: CheckoutService

Most frequently asked questions:

Q1. Strategy vs Factory?
Answer:
- Factory creates objects
- Strategy changes behavior/algorithm

Q2. Why is Strategy better than if-else?
Answer:
- each behavior is isolated in its own class
- easier to extend and test
- context code stays clean

Q3. When is Strategy a good fit?
Answer:
- when the same job can be done in multiple ways
- and the selected way may change at runtime

Best interview line:
"Strategy Pattern is used when there are multiple algorithms for the same task,
and I want to switch behavior without changing client code."

4) OBSERVER PATTERN
======================================================================

Code:
- desigen_pattern/4_observer/observer.py

One-line definition:
- Observer Pattern creates a one-to-many dependency so all subscribers get updated automatically when the subject changes.

When to use:
- event-driven systems
- notification systems
- live dashboards
- stock updates

Real-world example:
- YouTube channel notifies all subscribers when a new video is uploaded

In your code:
- Subject interface: Subject
- Concrete subject: YouTubeChannel
- Observer interface: Observer
- Concrete observers:
  - EmailSubscriber
  - MobileSubscriber

Most frequently asked questions:

Q1. Observer vs Pub-Sub?
Answer:
- Observer: subject usually knows observers directly
- Pub-Sub: publisher and subscriber are decoupled through broker/event bus

Q2. When should I use Observer?
Answer:
- when one object changes and many dependent objects should react automatically

Q3. What is a drawback of Observer?
Answer:
- too many observers can affect debugging and performance

Best interview line:
"Observer is useful when a change in one object should automatically notify many dependent objects."

5) DECORATOR PATTERN
======================================================================

Code:
- desigen_pattern/5_decorator/decorator.py

One-line definition:
- Decorator adds behavior to an object dynamically at runtime without changing its original class.

When to use:
- optional features
- runtime feature composition
- want to avoid subclass explosion

Real-world example:
- coffee with milk, sugar, whipped cream

In your code:
- Component: Coffee
- Concrete component: SimpleCoffee
- Base decorator: CoffeeDecorator
- Concrete decorators:
  - MilkDecorator
  - SugarDecorator
  - WhippedCreamDecorator

Most frequently asked questions:

Q1. Decorator vs Inheritance?
Answer:
- Inheritance is fixed at class level
- Decorator is flexible at runtime

Q2. Why use Decorator?
Answer:
- lets us combine features dynamically
- avoids too many subclasses like CoffeeWithMilkAndSugarAndCream

Q3. Real-world use of Decorator?
Answer:
- middleware
- logging wrappers
- authentication wrappers

Best interview line:
"Decorator wraps an object to add responsibilities dynamically without modifying the original class."

6) COMMAND PATTERN
======================================================================

Code:
- desigen_pattern/6_command/command.py

One-line definition:
- Command Pattern encapsulates a request as an object.

When to use:
- undo/redo support
- request queueing
- action history
- remote control or menu actions

Real-world example:
- remote control commands for light and fan

In your code:
- Command interface: Command
- Concrete commands:
  - LightOnCommand
  - LightOffCommand
  - FanStartCommand
  - FanStopCommand
- Receivers:
  - Light
  - Fan
- Invoker: RemoteControl

Most frequently asked questions:

Q1. What problem does Command solve?
Answer:
- decouples the sender of a request from the object that executes it

Q2. Why is Command useful for undo?
Answer:
- each command object stores the action and can implement reverse behavior in `undo()`

Q3. Command vs Strategy?
Answer:
- Strategy changes algorithm
- Command represents a request/action as an object

Best interview line:
"Command Pattern is useful when actions should be treated as objects so they can be stored, queued, logged, or undone."

7) ADAPTER PATTERN
======================================================================

Code:
- desigen_pattern/7_adopter/adopter.py

One-line definition:
- Adapter Pattern converts one interface into another interface that the client expects.

When to use:
- integrating third-party library
- working with legacy system
- incompatible interfaces

Real-world example:
- Checkout service expects `pay(amount)` but Stripe and PayPal expose different method names

In your code:
- Target interface: PaymentProcessor
- Adaptees:
  - StripeGateway
  - PayPalGateway
- Adapters:
  - StripeAdapter
  - PayPalAdapter
- Client: CheckoutService

Most frequently asked questions:

Q1. What problem does Adapter solve?
Answer:
- it lets incompatible classes work together without changing either side

Q2. Adapter vs Facade?
Answer:
- Adapter changes interface compatibility
- Facade gives a simplified interface to a subsystem

Q3. Where is Adapter commonly used?
Answer:
- third-party API integration
- legacy code integration
- wrapper libraries

Best interview line:
"Adapter acts like a translator between the client and an existing class with an incompatible interface."
